# Auditoría de readiness para producción — Hey Banco / Banregio

## Veredicto general

**NO LISTA PARA PRODUCCIÓN.**

La integración tiene una base técnica sólida para sandbox (cliente ADQ, actions, gateway
manager, checkout cableado, monitor admin, cancelación básica), pero presenta **bloqueadores
críticos** que impiden un go-live seguro con dinero real:

- Infraestructura de BD incompleta (migraciones faltantes).
- Sin 3D Secure Hey Banco implementado.
- Sin webhooks ni reconciliación automática robusta.
- Idempotencia insuficiente ante doble submit, reintentos y timeouts de red.
- Riesgo operativo alto en flujo **cobro → GDA → rollback** (laboratorio).
- Feature flag y modo sandbox activos por defecto.

**Estado recomendado hoy:** pruebas internas controladas con `HEYBANCO_ENABLED=true` en
staging, usando URLs y credenciales sandbox de Banregio Colecto. No activar en producción
hasta cerrar bloqueadores de Fase 1 y Fase 2 del plan de acción.

# Revisión de idempotencia

## Análisis del flujo actual

### Generación de referencia

```php
// ChargeHeyBancoTokenAction.php ~61
$reference = $reference ?? ('FM-' . $customer->id . '-' . time() . '-' . rand(1000, 9999));
```

- Nueva referencia en cada invocación si no se pasa explícitamente.
- `OrderAction` y `ChargePaymentMethodAction` **no pasan** referencia determinística.
- Colisión improbable pero **no imposible**; sin constraint UNIQUE.

### PaymentAttempt

- Se crea con `status=processing` antes de llamar a Banregio.
- No verifica si ya existe attempt `approved` para el mismo checkout/pedido.
- `token_id` apunta a `PaymentMethod.id` (correcto para Hey Banco).

### PaymentTransaction

- Se crea por cada llamada ADQ sin deduplicación.
- Folio generado con unicidad parcial (`HeyBancoClient::generateFolio` consulta `payment_transactions`).

### Transaction (negocio)

- Se crea solo si cobro approved.
- Sin unique en `reference_id` o `gateway_transaction_id`.

### Verify by reference / folio

- Solo en `handleTimeout()` cuando `BNRG_CODIGO_PROC=T`.
- No se usa como guardia idempotente antes de un segundo cobro.

### Retry en timeout

- Un solo intento de verify por referencia, luego por folio.
- Si ambos fallan → status `timeout` → excepción al usuario.
- No hay job de reintento.

### Frontend doble submit

- `LaboratoryCheckout.jsx`: `if (checkoutProcessing) return;` y botones deshabilitados con `checkoutProcessing`.
- Protección **parcial** (Inertia `processing`); no hay token anti-replay server-side.
- Refresh de página tras error puede reenviar formulario.

### FulfillLaboratoryCartOrderAction / OrderAction

- Si cobro OK y fulfillment falla → `RefundTransactionAction` en catch.
- Si usuario reintenta checkout completo → **nuevo cobro** con nueva referencia (doble cargo posible si refund no corrió).

## ¿Hay idempotencia suficiente?

**NO.**

## Dónde falta

1. Clave idempotente por sesión de checkout / intento de compra.
2. Constraint UNIQUE en referencia de cobro.
3. Verificación "ya cobrado" antes de segundo VENTA.
4. Reintento seguro tras timeout (verify-first, charge-never).
5. Backend idempotency middleware para `laboratory.checkout.store`.

## Cómo debería implementarse

1. Generar `idempotency_key` al entrar a confirmación checkout (UUID en `LaboratoryCheckoutDraft` o sesión).
2. Pasar esa clave como `BNRG_REF_CLIENTE1` o campo dedicado.
3. `UNIQUE(gateway, reference)` en `payment_attempts`.
4. En `ChargeHeyBancoTokenAction`: si existe attempt/transaction approved con misma key → retornar Transaction existente.
5. En timeout/error red: estado `pending_verification`, no re-cobrar; job verify.

## Constraints recomendados

- `payment_attempts`: UNIQUE(`reference`), INDEX(`customer_id`, `status`, `created_at`)
- `payment_transactions`: UNIQUE(`provider`, `reference`) WHERE reference IS NOT NULL
- `transactions`: INDEX(`gateway`, `gateway_transaction_id`)

## Pruebas a agregar

- Doble POST mismo checkout → un solo cargo Banregio (Http::fake assertSentCount 1).
- Timeout + verify approved → no segundo cargo en reintento.
- Timeout + verify failed → estado pending, no Transaction de negocio.


# Revisión de timeout y verificación

## Qué hace actualmente

1. `HeyBancoClient::post()` usa `Http::asForm()->timeout(HEYBANCO_TIMEOUT)`.
2. Si respuesta tiene `BNRG_CODIGO_PROC=T` → `handleTimeout()`.
3. `handleTimeout()`:
   - Si hay `referencia` en respuesta → `verifyAction->byReference()`.
   - Si no approved y hay `folio` → `verifyAction->byFolio()`.
   - Si verify approved → continúa flujo exitoso.
   - Si no → `paymentTransaction.status=timeout` → lanza `HeyBancoPaymentException`.

## VerifyHeyBancoTransactionAction

- Exige `$response->isApproved()` (codigo_proc=A).
- Para considerar verificación de venta exitosa usa `isVerificationApproved()` (A + proc_trans=A + estado_trans=C).
- Si verify no cumple → lanza excepción (no persiste estado intermedio útil para reconciliación).

## Si verify falla

- Excepción burbujea a `ChargeHeyBancoTokenAction` catch.
- `PaymentAttempt` → `status=error`.
- Usuario ve error de pago.
- **Si Banregio sí cobró:** dinero capturado sin `Transaction` de negocio ni pedido.

## Si falla conexión HTTP (no BNRG T)

- `\Throwable` en catch genérico.
- Mismo resultado: attempt error, sin verify automático.
- **Riesgo máximo de desincronización.**

## Si el banco cobró pero el sistema no lo sabe

- No hay job de reconciliación.
- Admin puede usar manualmente `POST payments/hey-banco/verify` (requiere auth usuario).
- BanregioMonitor muestra transacciones si se persistieron parcialmente.

## ¿Se necesita job de reconciliación?

**SÍ** — para producción.

Propuesta:
- Job `ReconcileHeyBancoPaymentJob` cada N minutos.
- Buscar `payment_transactions` con status `timeout`, `pending_verification`, o attempts `processing` > 5 min.
- Ejecutar verify; si approved, crear/recuperar `Transaction` y continuar fulfillment si aplica.

## ¿Pantalla admin de pagos pendientes?

# Revisión de reembolsos y cancelaciones

## Qué está implementado

| Componente | Comportamiento |
|------------|----------------|
| `RefundTransactionAction::refund()` | Detecta gateway `hey_banco`; verifica `refunded_at` |
| `refundHeyBancoTransaction()` | Llama `HeyBancoPaymentGateway::refundOrCancel()` |
| `CancelHeyBancoTransactionAction` | Resuelve referencia desde `details.payment_details.banregio_reference`, `gateway_transaction_id`; `cancelByReference()` con monto total |
| Persistencia | Nueva `PaymentTransaction` flow=cancellation; actualiza `Transaction.refunded_at`, `gateway_status=refunded`, `details.cancellation` |
| `RefundTransactionAction::__invoke` | Si refund OK → `$transaction->delete()` (soft delete) |

## Referencia Banregio

Resolución en orden:
1. `details.payment_details.banregio_reference`
2. `gateway_transaction_id`
3. `details.payment_details.reference` (referencia cliente FM-*, no referencia Banregio — fallback débil)

## Reembolso total vs parcial

- Solo **cancelación total** (`amountCents / 100` completo).
- Reembolso parcial: **no implementado**.

## Si cancelación falla

- `refundHeyBancoTransaction` retorna `false`.
- `Transaction` **no** se marca `refunded_at`.
- `__invoke` no elimina Transaction.
- Log error; sin notificación a ops.

## Si admin cancela compra

- Depende de que el flujo admin invoque `RefundTransactionAction` con Transaction que tenga `gateway=hey_banco` y referencia Banregio válida.
- Si `gateway_transaction_id` es folio en lugar de referencia Banregio, cancelación puede fallar.

## Si GDA falla después del cobro (laboratorio)

- `OrderAction` catch → `RefundTransactionAction($transaction)`.
- Si cancelación Banregio exitosa → Transaction eliminada.
- Si cancelación falla → cliente cobrado, sin pedido, Transaction persiste.

## Si se intenta reembolsar dos veces

- Segunda llamada: `refunded_at` ya seteado → `refund()` retorna `false` (log warning).
- Protección **suficiente** a nivel aplicación si `refunded_at` se setea correctamente en primer éxito.

## Riesgos

1. Cancelación fallida silenciosa post-GDA.
2. `__invoke` borra Transaction tras refund — dificulta auditoría histórica (soft delete mitiga parcialmente).
3. Sin verify post-cancelación.
4. Sin reintento automático.

## Recomendaciones

1. Estado `refund_pending` / `refund_failed` en Transaction.
2. No soft-delete hasta confirmar cancelación + verify.
3. Cola de reintentos cancelación.
4. Alerta admin si refund falla tras fallo GDA.

# Resumen ejecutivo

1. La integración Hey Banco / Banregio Colecto **NO está lista para producción** en su estado actual.
2. Existe una base funcional en sandbox: cliente ADQ, tokenización, cobro, verify, cancelación y checkout lab/membresía cableados.
3. Los **bloqueadores críticos** son: migraciones BD faltantes, ausencia de 3DS, sin reconciliación robusta, idempotencia insuficiente y riesgo cobro→GDA→refund.
4. Activar hoy en producción con dinero real sería **altamente riesgoso** (fallos BD, doble cargo, cobros huérfanos, cancelaciones fallidas).
5. `HEYBANCO_ENABLED=false` y `HEYBANCO_MODE=AUT` por defecto confirman que el proyecto no contempla prod aún.
6. La farmacia **no** usa Hey Banco; solo laboratorio y membresía médica.
7. Admin tiene monitor Banregio útil pero incompleto para operar incidentes de producción.
8. Solo 2 tests unitarios cubren Hey Banco; faltan feature, integración y E2E.
9. **Primero:** crear migraciones `payment_methods` y `payment_transactions` y validar en BD limpia.
10. **Segundo:** idempotencia + job reconciliación + manejo timeout/red.
11. **Tercero:** confirmar requisito 3DS con Banregio e implementar si es obligatorio.
12. **Cuarto:** pruebas E2E lab/membresía/refund/GDA-fail en staging con credenciales sandbox.
13. **Quinto:** cerrar gaps admin (filtros, PaymentDetails verify) y checklist go-live.
14. Solo después de Fases 1–3 con sign-off QA/Ops considerar `HEYBANCO_ENABLED=true` en producción con `MODE=PRD`.
15. Migración total desde EfevooPay (Fase 5) requiere re-tokenización de tarjetas y plan de comunicación a usuarios.
