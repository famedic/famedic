# Causa exacta del fallo

La migración `database/migrations/2025_06_27_205929_migrate_subscription_invoices_from_old_db.php`
consultaba inmediatamente la conexión `mysqlold` sin validar si la base legacy existe:

    DB::connection('mysqlold')->table('subscription_invoices')->chunk(...)

En ambientes sin `famedic_old` (Docker local, testing, CI, migrate:fresh):

    SQLSTATE[HY000] [1049] Unknown database 'famedic_old'
    (Connection: mysqlold, SQL: select * from `subscription_invoices` ...)

La conexión `mysqlold` está definida en `config/database.php` con `database => 'famedic_old'`
hardcodeado; el host/credenciales son los mismos que `mysql`, pero la base antigua no se
provisiona en entornos nuevos.


# Archivo corregido

| Archivo | Cambio |
|---------|--------|
| `database/migrations/2025_06_27_205929_migrate_subscription_invoices_from_old_db.php` | Guard `legacySourceIsAvailable()` antes del chunk; no-op seguro si legacy no existe |


# Análisis de la migración original

| Aspecto | Detalle |
|---------|---------|
| Conexión legacy | `mysqlold` (`config/database.php`, BD `famedic_old`) |
| Tabla legacy leída | `subscription_invoices` (+ `transactionables`, `transactions` por factura) |
| Tablas destino (conexión `mysql`) | `medical_attention_subscriptions`, `transactions`, `transactionables` |
| Modelos Eloquent | Ninguno (usa `DB::table` directamente) |
| Operación | **Crea** suscripciones nuevas y transacciones nuevas a partir de datos legacy |
| Rollback (`down`) | No-op documentado (migración de datos sin reversa) |

Flujo cuando el origen existe:
1. Lee `subscription_invoices` de `famedic_old` en chunks de 100.
2. Por cada factura, verifica que el `customer_id` exista en la BD nueva.
3. Inserta en `medical_attention_subscriptions`.
4. Migra transacciones asociadas (`transactionables` + `transactions`) con metadata en `details`.
5. Omite facturas cuyo cliente no existe (warning en log).


# Cómo se validó la conexión legacy

Método privado `legacySourceIsAvailable()`:

1. **Configuración:** `config('database.connections.mysqlold')` debe existir.
2. **Conexión PDO:** `DB::connection('mysqlold')->getPdo()` dentro de try/catch.
3. **Base inexistente:** si el error es código MySQL **1049** o mensaje `Unknown database`,
   se considera origen no disponible → no-op con `Log::warning()`.
4. **Otros errores de conexión** (credenciales inválidas con BD existente, timeouts, etc.):
   se **re-lanzan** para no ocultar fallos reales en producción.


# Cómo se validó la tabla legacy

Tras conectar exitosamente:

    Schema::connection('mysqlold')->hasTable('subscription_invoices')

Si la tabla no existe → `Log::warning()` y no-op.


# Qué pasa si `famedic_old` no existe

- La migración termina con **éxito** (no lanza excepción).
- Registra warning: `Skipping subscription invoices legacy migration: legacy database is not available.`
- No inserta datos en la BD nueva.
- `migrate:fresh` puede continuar con migraciones posteriores.
- Tests con `RefreshDatabase` no fallan por esta migración.


# Qué pasa si `famedic_old` sí existe

- Se valida conexión y tabla.
- Se ejecuta el **mismo comportamiento original** (chunk + creación de suscripciones y transacciones).
- Errores de datos reales (FK, columnas, permisos con BD accesible) **no se silencian**.


# Criterio: errores que NO se ocultan

| Situación | Comportamiento |
|-----------|----------------|
| BD `famedic_old` no existe (1049) | No-op + warning |
| Tabla `subscription_invoices` ausente en legacy | No-op + warning |
| Conexión `mysqlold` no configurada | No-op + warning |
| BD legacy accesible pero fallo en INSERT/SELECT de datos | **Excepción** (comportamiento original) |
| Cliente no encontrado en BD nueva | Warning + skip de esa factura (comportamiento original) |


# Impacto en producción

- Entornos con `famedic_old` provisionada: **sin cambio funcional**; migración de datos intacta.
- Entornos nuevos sin legacy: dejan de fallar; no se crean datos ficticios.
- No se borran datos históricos ni se elimina la migración.
- No se requiere crear `famedic_old` en local/testing/CI.


# Impacto en testing/CI

- `RefreshDatabase` ya no falla en `2025_06_27_205929_...` por `Unknown database`.
- La migración hace no-op en ~10 ms cuando no hay legacy.
