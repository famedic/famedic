# Resumen general de la integración Hey Banco / Banregio

Se ha implementado una integración con **Banregio Colecto** (plataforma ADQ de Hey Banco /
Banregio) para **tokenización de tarjetas** y **cobros con token**, usando el protocolo BNRG_
(form POST con respuesta en headers HTTP).

## Qué cubre el flujo actual

1. **Tokenización directa** (comando CREACION_TOKEN) sin 3D Secure.
2. **Cobro con token** (comando VENTA) contra tarjetas previamente guardadas.
3. **Verificación** de transacciones (comando VERIFICACION) por referencia o folio — usada
   principalmente para recuperación ante timeout.
4. **Cancelación** (comando CANCELACION) por referencia de venta original — usada como
   mecanismo de reembolso.
5. **Abstracción de gateway** vía PaymentGatewayInterface + PaymentGatewayManager, conviviendo
   con EfevooPay.
6. **Monitoreo administrativo** dedicado (Banregio Monitor) para tokens, transacciones ADQ e
   intentos de pago.
7. **API JSON autenticada** para pruebas manuales (tokenize / charge / verify).
8. **Conexión al checkout de laboratorio** y **suscripción médica** a través del mismo
   ChargePaymentMethodAction que usa EfevooPay.

## Qué parece completo

- Cliente HTTP (HeyBancoClient) con operaciones ADQ.
- Wrapper de respuesta (HeyBancoResponse) con códigos BNRG_CODIGO_PROC.
- Persistencia en payment_methods, payment_transactions, payment_attempts y transactions.
- Tokenización desde UI de métodos de pago (PaymentMethods/Create).
- Cobro en flujos de negocio que usan ChargePaymentMethodAction.
- Cancelación/reembolso vía RefundTransactionAction.
- Tests unitarios del cliente y de cancelación.
- UI admin Banregio + badges frontend para hey_banco.

## Qué parece pendiente o incompleto

- **3D Secure Hey Banco**: variables y URL configuradas, pero sin implementación de flujo.
- **Webhooks/callbacks** entrantes desde Banregio: no existen.
- **Migraciones versionadas** para tablas payment_methods y payment_transactions: no
  encontradas en database/migrations/.
- **Feature flag desactivado por defecto** (HEYBANCO_ENABLED=false).
- **Farmacia en línea**: no integrada (sigue en Stripe/Odessa).
- **Filtros admin de compras de laboratorio**: no incluyen hey_banco en validación de filtro.
- **PaymentLog**: modelo existe pero Hey Banco no lo utiliza.
- **Paridad completa con EfevooPay**: webhooks, 3DS, API pública documentada, simulación mock.


# Estado actual de la implementación

**Veredicto: parcial — funcional en sandbox bajo feature flag — lista para pruebas internas
— NO lista para producción — pendiente de conectar con checkout real en ambiente productivo.**

## Justificación basada en código

| Indicador | Evidencia |
|-----------|-----------|
| En desarrollo / parcial | HEYBANCO_ENABLED default false; 3DS no implementado; migraciones faltantes |
| Sandbox | adq_url default apunta a testcolecto.banregio.com; HEYBANCO_MODE default AUT |
| Lista para pruebas | Rutas API + UI tokenización + tests Http::fake + tarjetas test en config |
| No producción | Sin webhooks, sin 3DS, flag off, modo AUT documentado solo para sandbox |

La integración **sí está cableada** al checkout de laboratorio y membresía médica, pero solo
**cuando HEYBANCO_ENABLED=true** y el usuario tiene tarjetas hey_banco:* tokenizadas.

# Servicios implementados

## 1. HeyBancoClient

- **Clase:** App\Services\Payments\HeyBanco\HeyBancoClient
- **Archivo:** app/Services/Payments/HeyBanco/HeyBancoClient.php
- **HTTP:** Laravel Http facade (asForm, timeout configurable)

### Métodos públicos

| Método | Qué hace | Endpoint | Entrada | Retorno |
|--------|----------|----------|---------|---------|
| createToken($cardData, $reference?) | CREACION_TOKEN | POST adq_url | card_number, exp_month, exp_year, cvv | HeyBancoResponse |
| chargeToken($token, $amount, $reference, $metadata?) | VENTA | POST adq_url | token, monto, referencia cliente | HeyBancoResponse |
| verifyByReference($reference, $mediaId?) | VERIFICACION | POST adq_url | BNRG_REF_TRANS_PREVIA | HeyBancoResponse |
| verifyByFolio($folio, $mediaId?) | VERIFICACION | POST adq_url | BNRG_FOLIO | HeyBancoResponse |
| cancelByReference($reference, $mediaId?, $amount?) | CANCELACION | POST adq_url | BNRG_REF_TRANS_PREVIA, monto opcional | HeyBancoResponse |
| normalizeHeaders($headers) | Parsea headers BNRG_ | — | headers HTTP | array normalizado |
| generateFolio() | Folio único FM+hex (max 12 chars) | — | — | string |
| localDate() | Fecha DDMMAAAA zona Monterrey | — | — | string |
| localTime() | Hora HHMMSS zona Monterrey | — | — | string |

### Manejo de errores

- No lanza excepciones HTTP directamente; encapsula en HeyBancoResponse.
- Códigos de negocio en BNRG_CODIGO_PROC: A=approved, D=declined, R=rejected, T=timeout, X=failed.
- Las Actions interpretan isApproved(), isTimeout(), etc.

# Actions implementadas

## CreateHeyBancoTokenAction

- **Archivo:** app/Actions/Payments/HeyBanco/CreateHeyBancoTokenAction.php
- **Responsabilidad:** Tokenizar tarjeta en Colecto y persistir PaymentMethod
- **Entrada:** User $user, array cardData (card_number, exp_month, exp_year, cvv, card_holder?, alias?)
- **Servicios:** HeyBancoClient::createToken
- **Modelos creados:** PaymentTransaction (flow=token_creation), PaymentMethod (provider=hey_banco)
- **Retorno:** PaymentMethod
- **Excepciones:** HeyBancoPaymentException si no approved o sin token
- **Transacción DB:** Sí, envuelta en DB::transaction

## ChargeHeyBancoTokenAction

- **Archivo:** app/Actions/Payments/HeyBanco/ChargeHeyBancoTokenAction.php
- **Responsabilidad:** Cobrar monto con token guardado
- **Entrada:** Customer, amountCents, paymentMethodId (hey_banco:{id}), reference?
- **Servicios:** HeyBancoClient::chargeToken, VerifyHeyBancoTransactionAction (si timeout)
- **Modelos:** PaymentAttempt (processing→approved/declined/error), PaymentTransaction (token_charge),
  Transaction (negocio), actualiza PaymentMethod.last_used_at
- **Retorno:** Transaction
- **Excepciones:** HeyBancoPaymentException (método inválido, expirado, rechazado, error genérico)
- **Timeout:** Si codigo_proc=T, intenta verify por referencia o folio antes de fallar

## VerifyHeyBancoTransactionAction

- **Archivo:** app/Actions/Payments/HeyBanco/VerifyHeyBancoTransactionAction.php
- **Métodos:** byReference(), byFolio()
- **Modelos:** PaymentTransaction (flow=verification); actualiza previousTransaction a approved si OK
- **Excepciones:** HeyBancoPaymentException si verificación no approved

## CancelHeyBancoTransactionAction

- **Archivo:** app/Actions/Payments/HeyBanco/CancelHeyBancoTransactionAction.php
- **Entrada:** Transaction (negocio), Customer opcional
- **Resuelve referencia desde:** details.payment_details.banregio_reference, gateway_transaction_id
- **Modelos:** PaymentTransaction (flow=cancellation, previous_reference)
- **Excepciones:** HeyBancoPaymentException si sin referencia

## ChargePaymentMethodAction (unificador)

- **Archivo:** app/Actions/Payments/ChargePaymentMethodAction.php
- **Responsabilidad:** Resolver gateway y ejecutar charge
- **Usado por:** OrderAction (lab), PurchaseRegularSubscriptionAction (membresía)

## Modelos reutilizados

### Transaction (tabla: transactions)

Al cobrar Hey Banco se crea Transaction con:
- payment_method = hey_banco
- gateway = hey_banco
- gateway_transaction_id = BNRG_REFERENCIA (o folio fallback)
- gateway_status = completed
- gateway_response = HeyBancoResponse::toArray()
- gateway_token = hb-token-ref:{hash parcial} (NO es el token real de Colecto)
- gateway_authorization_code = BNRG_CODIGO_AUT
- details JSON estructurado:
  - token_info: payment_method_id, card_brand, card_last_four, alias
  - payment_details: banregio_reference, authorization_code, folio, payment_attempt_id, payment_transaction_id
  - customer_info: customer_id, user_id

No existen para Hey Banco. Efevoo3dsSession es exclusivo EfevooPay.


# Flujo de tokenización

## Implementado (sin 3DS)

1. **Pantalla:** PaymentMethods/Create.jsx — radio "Hey Banco / Banregio Colecto" si heyBancoEnabled
2. **Submit:** POST payment-methods.store con payment_provider=hey_banco
3. **Controller:** PaymentMethodController::storeHeyBancoCard
4. **Gateway:** PaymentGatewayManager::forProvider('hey_banco')->tokenize()
5. **Action:** CreateHeyBancoTokenAction → HeyBancoClient::createToken (CREACION_TOKEN)
6. **Respuesta procesador:** Headers BNRG_TOKEN, BNRG_REFERENCIA, BNRG_CODIGO_PROC
7. **Persistencia:**
   - PaymentTransaction (flow=token_creation)
   - PaymentMethod (provider_token = BNRG_TOKEN)
8. **UI posterior:** PaymentMethods index muestra tarjeta con id hey_banco:{id}
9. **Checkout:** Customer.paymentMethods() incluye hey_banco:* → seleccionable en PaymentMethodStep

## Alternativa API (pruebas)

POST /payments/hey-banco/tokenize → mismo CreateHeyBancoTokenAction

## Qué falta

- Flujo 3DS Hey Banco (config HEYBANCO_3DS_* sin código)
- Validación de expiración de tarjeta en storeHeyBancoCard (Efevoo sí valida; Hey Banco delega al procesador)
- Endpoint de eliminación remota del token en Colecto (solo status=inactive local)


# Flujo de pago

## Flujo completo (laboratorio / membresía)

1. **Selección método:** PaymentMethodStep muestra tarjetas de customer.paymentMethods() incluyendo hey_banco:N
2. **Validación checkout:** StoreLaboratoryPurchaseRequest valida que payment_method esté en lista permitida
3. **Creación intento:** ChargeHeyBancoTokenAction crea PaymentAttempt status=processing
4. **Llamada procesador:** HeyBancoClient::chargeToken (VENTA) con BNRG_TOKEN, BNRG_MONTO_TRANS, BNRG_REF_CLIENTE1
5. **Respuesta exitosa:** BNRG_CODIGO_PROC=A → PaymentTransaction approved
6. **Transaction negocio:** Transaction creada con gateway=hey_banco y details completos
7. **Creación compra:**
   - Lab: FulfillLaboratoryCartOrderAction + LaboratoryPurchase
   - Membresía: CreateRegularSubscriptionAction + attach transaction
8. **Fulfillment:** Post-pago; si falla fulfillment → RefundTransactionAction automático
9. **Notificaciones:** LaboratoryPurchaseCreated (lab); no específicas Hey Banco
10. **Redirección:** laboratory-purchases.show con mensaje éxito
11. **Errores:** HeyBancoPaymentException → redirect back con error payment_method

## Conectado con

| Módulo | Estado |
|--------|--------|
| Laboratorio | SÍ — vía OrderAction + ChargePaymentMethodAction |
| Farmacia | NO — Stripe/Odessa únicamente |
| Membresía médica | SÍ — vía PurchaseRegularSubscriptionAction |
| Checkout general | Parcial — solo flujos que usan ChargePaymentMethodAction |
| Pruebas aisladas | SÍ — API payments/hey-banco/* |

## Orden de operaciones y GDA (laboratorio)

El cobro ocurre ANTES de FulfillLaboratoryCartOrderAction (incluye GDA). Si fulfillment falla,
RefundTransactionAction intenta cancelar en Banregio. Riesgo: cobro exitoso + GDA fallido +
cancelación fallida = inconsistencia.

# Riesgos técnicos detectados

1. **Tokens no portables:** provider_token Colecto atado a afiliación/media_id; no interoperable con Efevoo
2. **Checkout parcialmente conectado:** funciona pero flag off por defecto; UI checkout no distingue proveedor
3. **Falta idempotencia:** referencia FM-{customer}-{time}-{rand} sin deduplicación explícita
4. **Falta webhook:** estados finales dependen de respuesta síncrona o verify manual
5. **Timeout handling limitado:** verify automático solo en ChargeHeyBancoTokenAction, no en otros flujos
6. **Migraciones faltantes:** payment_methods/payment_transactions pueden no existir en BD nueva
7. **gateway_token en Transaction:** guarda hash referencial, no token real — puede confundir debugging
8. **Nombres inconsistentes:** hey_banco vs Banregio vs Hey Banco en UI
9. **Admin filtros:** laboratory purchases no filtran hey_banco
10. **Impacto GDA:** cobro antes de fulfillment; rollback depende de cancelación Banregio exitosa
11. **3DS obligatorio en producción:** sin implementar, puede bloquear go-live bancario
12. **HEYBANCO_MODE=AUT:** aprobar siempre en sandbox; riesgo si se despliega a prod sin cambiar a PRD
13. **Affiliation/media IDs en .env.example:** valores de sandbox expuestos como ejemplo (no secretos pero sensibles operativamente)
14. **PaymentAttempt.token_id:** FK lógica a PaymentMethod.id para Hey Banco pero FK a efevoo_tokens para Efevoo — semántica mixta
15. **EfevooPaymentGateway.refundOrCancel:** retorna supported=true sin ejecutar reembolso real — deuda técnica paralela
