1. OBJETIVO DEL FLUJO (CAMINO B)
================================================================================

Cobro 3D Secure usando una tarjeta YA TOKENIZADA en Famedic:

- El usuario tiene un PaymentMethod con provider=hey_banco y provider_token guardado.
- En checkout NO se envían PAN, expiración ni CVV a Banregio.
- Se envía BNRG_TOKEN al endpoint 3DS de Colecto Banregio.
- Banregio autentica al tarjetahabiente (challenge 3DS).
- Si la autenticación es exitosa, Banregio YA COBRA en su motor Colecto (no en ADQ).
- Banregio devuelve el resultado a Famedic vía POST a BNRG_URL_RESPUESTA.
- Famedic valida firma, guarda trazabilidad y continúa fulfillment (pedido laboratorio).

REGLA CRÍTICA: Si BNRG_CODIGO_PROC=A, NO volver a llamar ChargeHeyBancoTokenAction
ni chargeToken del cliente ADQ. Eso duplicaría el cargo.

4. FLUJO PASO A PASO
================================================================================

1. Paciente selecciona tarjeta Hey Banco guardada (payment_method = hey_banco:{id})
2. Confirma compra en checkout laboratorio
3. OrderAction valida totales/cupón/cita
4. Si amount > 0 y HEYBANCO_3DS_ENABLED=true y método es hey_banco:*:
   a. InitiateHeyBanco3dsLaboratoryCheckoutAction crea sesión + inicia 3DS
   b. Lanza HeyBanco3dsRedirectRequiredException con redirect_url
5. LaboratoryPurchaseController redirige a payments.hey-banco.3ds.redirect
6. HeyBanco3dsRedirect.jsx muestra "Redirigiendo..." y hace window.location al URL Banregio
7. Usuario completa challenge 3DS en Banregio
8. Banregio POSTea a BNRG_URL_RESPUESTA (= route payments.hey-banco.3ds.callback)
9. HandleHeyBanco3dsCallbackAction:
   - Busca sesión por BNRG_FOLIO
   - Valida BNRG_HASH si HEYBANCO_3DS_SECURE_API=true
   - Actualiza Payment3dsSession, PaymentAttempt, PaymentTransaction
   - Mapea BNRG_CODIGO_PROC
10. Si A (aprobado):
    - Crea Transaction de negocio (gateway=hey_banco)
    - Ejecuta FulfillLaboratoryHeyBanco3dsPaymentAction (pedido GDA)
    - NO llama ChargeHeyBancoTokenAction
11. Usuario ve HeyBanco3dsResult.jsx o es redirigido al pedido

Si HEYBANCO_3DS_ENABLED=false:
- OrderAction usa flujo normal → ChargePaymentMethodAction → HeyBancoPaymentGateway → chargeToken ADQ

6. REGISTROS RELACIONADOS EN COBRO 3DS
================================================================================

Al iniciar sesión se crean:

PaymentAttempt:
  gateway=hey_banco, status=pending_3ds

PaymentTransaction:
  provider=hey_banco, flow=token_3ds_charge, status=pending_3ds

Payment3dsSession:
  status=pending → redirect_required (tras start exitoso)

Tras callback aprobado:

PaymentAttempt → approved
PaymentTransaction → approved (con BNRG_REFERENCIA, BNRG_CODIGO_AUT, etc.)
Payment3dsSession → approved
Transaction (tabla transactions) → nueva fila con details.payment_details.payment_3ds_session_id

================================================================================
7. PAYLOAD DE INICIO 3DS (HeyBanco3dsClient)
================================================================================

POST application/x-www-form-urlencoded a HEYBANCO_3DS_URL

Campos enviados:
  BNRG_ID_MEDIO          → config heybanco.3ds_media_id
  BNRG_ID_AFILIACION     → config heybanco.3ds_affiliation
  BNRG_MONTO_TRANS       → formato ####.## (ej. 150.00)
  BNRG_TOKEN             → payment_methods.provider_token
  BNRG_URL_RESPUESTA     → route('payments.hey-banco.3ds.callback')
  BNRG_FOLIO             → folio de la sesión
  BNRG_REF_CLIENTE1      → referencia interna
  BNRG_MODO_TRANS        → config heybanco.mode (AUT sandbox, PRD producción)
  BNRG_HORA_LOCAL        → HHMMSS zona America/Monterrey
  BNRG_FECHA_LOCAL       → DDMMAAAA zona America/Monterrey

Si HEYBANCO_3DS_SECURE_API=true:
  BNRG_MODO_API_SEC=true
  BNRG_HASH              → firma generada

Campos que NO se envían (Camino B):
  BNRG_NUMERO_TARJETA
  BNRG_FECHA_EXP
  BNRG_CODIGO_SEGURIDAD

Respuesta esperada:
  URL de redirección en BNRG_URL_REDIRECCION, BNRG_REDIRECT_URL o body HTML

================================================================================
8. CALLBACK — CAMPOS BNRG_* RELEVANTES
================================================================================

Banregio POSTea a BNRG_URL_RESPUESTA. Campos que procesa Famedic:

  BNRG_3DS_ECI, BNRG_3DS_UCAF, BNRG_3DS_XID
  BNRG_CODIGO_PROC, BNRG_CODIGO_AUT, BNRG_CODIGO_EMISOR
  BNRG_TEXTO, BNRG_REFERENCIA, BNRG_ID_MEDIO, BNRG_ID_AFILIACION
  BNRG_MONTO_TRANS, BNRG_FOLIO, BNRG_REF_CLIENTE1, BNRG_HASH

Mapeo BNRG_CODIGO_PROC:
  A → approved
  D → declined
  R → rejected
  T → timeout
  X → failed

================================================================================
9. FIRMA BNRG_HASH (HeyBanco3dsSignatureService)
================================================================================

Algoritmo implementado:
  HMAC-SHA256 + Base64
  Secret: HEYBANCO_3DS_SECRET_KEY (config heybanco.3ds_secret_key)
  Valores normalizados con rawurldecode antes de concatenar

Request — orden de campos en config/heybanco.php → 3ds_request_sign_fields:
  BNRG_ID_MEDIO, BNRG_ID_AFILIACION, BNRG_MONTO_TRANS, BNRG_TOKEN,
  BNRG_FOLIO, BNRG_REF_CLIENTE1, BNRG_MODO_TRANS, BNRG_FECHA_LOCAL, BNRG_HORA_LOCAL

Response — orden en 3ds_response_sign_fields:
  BNRG_CODIGO_PROC, BNRG_CODIGO_AUT, BNRG_CODIGO_EMISOR, BNRG_REFERENCIA,
  BNRG_FOLIO, BNRG_MONTO_TRANS, BNRG_3DS_ECI, BNRG_3DS_UCAF, BNRG_3DS_XID

⚠️ TODO PENDIENTE: Confirmar orden exacto con Manual Colecto 3DS V1.2.
   Los campos son configurables en config/heybanco.php sin cambiar código.

14. checkout_context (JSON en Payment3dsSession)
================================================================================

Guardado al iniciar checkout laboratorio para fulfillment post-callback:

{
  "type": "laboratory_checkout",
  "customer_id": ...,
  "address_id": ...,
  "contact_id": ...,
  "laboratory_brand": "azteca|...",
  "total_cents": ...,
  "amount_charged_cents": ...,
  "discount_cents": ...,
  "coupon_id": null|int,
  "laboratory_appointment_id": null|int,
  "reference": "FM-LAB-..."
}

FulfillLaboratoryHeyBanco3dsPaymentAction usa este contexto para:
- Validar totales del carrito
- Resolver paciente (cita o contacto)
- Llamar FulfillLaboratoryCartOrderAction
- Si fulfillment falla: intenta RefundTransactionAction, marca refund_pending si falla
