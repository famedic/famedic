# Causa exacta del fallo

La migración `database/migrations/2026_05_15_120000_add_azteca_rmn_de_pie_simple_laboratory_test.php`
insertaba el estudio con:

    laboratory_test_category_id => 9   (hardcodeado)

En `migrate:fresh` sobre una base limpia:

- `laboratory_test_categories` solo tiene categorías creadas por migraciones previas
  (p. ej. «Chequeos y Paquetes» en `2025_04_21_modify_laboratory_tests_table`).
- No existe fila con `id = 9`.
- El `updateOrCreate` violaba la FK → `SQLSTATE[23000] Integrity constraint violation: 1452`.


# Archivo corregido

| Archivo | Cambio |
|---------|--------|
| `database/migrations/2026_05_15_120000_add_azteca_rmn_de_pie_simple_laboratory_test.php` | Resolver categoría por nombre; mantener `updateOrCreate` por `gda_id` |


# Por qué fallaba el ID 9

El ID 9 no es un identificador de negocio estable: en producción es el autoincrement
que le tocó a la categoría **«Resonancia»** cuando se importaron los estudios vía
`LaboratoryTestsImport` (columna 6 = nombre de categoría).

Evidencia:

- Índice Algolia (catálogo de producción): `gda_id` **445052** (`RMN DE PIE SIMPLE`,
  marca Azteca) tiene `category: "Resonancia"`.
- La migración original asignaba `laboratory_test_category_id => 9` al mismo `gda_id`.
- Mismo patrón que el fix del ID **12** → **«Chequeos y Paquetes»**.


# Nombre de la categoría correcta

**Resonancia**

(Confirmado por catálogo Algolia y estudios RMN del mismo tipo; en MySQL local
post-partial-migrate no existía `id = 9` porque no hay import de producción.)


# Cómo se resolvió la categoría

```php
$resonanceCategoryId = LaboratoryTestCategory::firstOrCreate([
    'name' => 'Resonancia',
])->id;
```

- Si la categoría ya existe (producción): se reutiliza sin duplicar.
- Si no existe (fresh): se crea con el nombre correcto.
- Se usa `$resonanceCategoryId` en lugar del literal `9`.
