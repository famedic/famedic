# Causa exacta del fallo original

La migración `database/migrations/2025_01_08_140518_create_tax_profiles_table.php`
estaba **mal implementada**:

- El nombre sugiere `CREATE tax_profiles`.
- El código ejecutaba `Schema::table('tax_profiles', ...)` (ALTER).
- **No existía** ninguna otra migración con `Schema::create('tax_profiles')`.
- En `migrate:fresh`, la tabla no existe → error:
  `SQLSTATE[42S02]: Table 'tax_profiles' doesn't exist` (MySQL)
  o `no such table: tax_profiles` (SQLite en tests).

Además, `database/migrations/2025_12_09_002225_add_columns_tax_profiles_table.php`
duplicaba el mismo ALTER (mismas columnas), lo que habría causado error de
columna duplicada tras corregir el create.


# Archivos corregidos

| Archivo | Cambio |
|---------|--------|
| `database/migrations/2025_01_08_140518_create_tax_profiles_table.php` | Reescrita: `Schema::create('tax_profiles')` con todas las columnas; `hasTable` guard para bases que ya tengan la tabla |
| `database/migrations/2025_12_09_002225_add_columns_tax_profiles_table.php` | Reescrita: ALTER idempotente con `hasColumn` por columna (compatibilidad bases existentes) |


# Criterio de corrección

- La migración `2025_01_08` nunca creó la tabla en repos limpios → **seguro reemplazar
  el contenido** por un `create` completo (no es alteración de producción ya aplicada
  de forma exitosa en BD vacías).
- `if (Schema::hasTable('tax_profiles')) return;` protege entornos donde la tabla ya
  fue creada manualmente o por despliegues previos.
- La migración `2025_12_09` se conserva como **parche histórico idempotente** para
  BD que tengan solo columnas base sin las de extracción automática.
